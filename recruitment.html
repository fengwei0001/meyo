龙虾招聘大会 · 活动策划与产品验证方案

以一场爆点活动为觅游「人 + Agent 撮合网络」做冷启动，并以「招聘」作为第一个验证场景
日期：2026 年 6 月 ｜ 初步方案，供团队讨论 ｜ 拟定共同 owner：产品 + 招聘/HR + 法务

一、一句话定位

一场面向大学生的「龙虾招聘大会」——让你的 AI（龙虾）帮你找实习，让企业用 AI 找到真正会用 AI 的年轻人。

它同时是三件事：

  1. 一场能「爆」的增长活动；
  2. 觅游撮合网络「招聘」场景的 MVP 验证；
  3. 撮合双边市场的 冷启动（用美团自有岗位坐满供给侧）。

二、为什么是这场活动：多方共赢 + 时机俱备

参与方痛点 / 诉求这场活动给到的
学生找实习难；不会包装自己；不知道机会在哪一只「求职龙虾」替自己持续找机会、包装简历、主动撮合
企业（先用美团）急于找到 AI native 的年轻人——他们才是未来用 AI 高效初筛，精准锁定「真的会用 AI」的候选人
觅游 / 平台流量低、没冷启动、功能没人用没反馈一次爆点事件带来真实供需，点着撮合网络第一块试验田

时机三连击：

  • 全国约 4800 万在校大学生，存量巨大；
  • 暑期临近，实习需求集中爆发；
  • AI native 人才正是当下企业最焦虑、最想抢的群体。

这三项叠加，活动具备「天然会爆」的传播势能。

三、关键认知：别把它办成「带龙虾皮肤的招聘会」

这是整份方案最重要的一句话，也是我们必须时刻自检的红线：

同样一波流量，它可以只是「办了一场活动」，也可以是「验证了撮合网络假设」——两者价值天差地别。

上一轮我们已经得出教训：moltbook 正是被一波新鲜感流量冲上 22M、又跌回 1.7M。 招聘大会如果设计不当，本质上也是一次性新鲜感事件。要让它落在「验证产品」这一侧，必须卡住下面四个设计原则。

四、四个必须卡住的设计原则

原则 1：撮合动作必须由 Agent 来做

真正要验证的核心假设是——「人愿不愿意托管一个代理人 Agent，让 AI 替自己去撮合」。因此活动机制应是：学生的「求职龙虾」 ↔ 美团的「招聘龙虾」，由 Agent 完成简历包装、初筛、匹配、破冰，最后把合适的人推给真人面试。

❌ 反例：只是传简历进系统 → 那是 HR 在办活动，与产品无关。

原则 2：必须有「活动后留存」设计

活动当天即流量峰值，之后必然回落。要提前设计：学生的「求职龙虾」在活动后继续活着——继续帮他匹配别的实习、别的雇主、跟进投递。

这样觅游就从「一场活动」升级为「每个年轻人的求职分身」，这才是留存的根，也是与 moltbook 划清界限的地方。

原则 3：供给侧先用美团，但要讲清这是「第一期」

用美团岗位做锚定租户启动没问题，但要明确终态需要多雇主。

  • 一期：美团自有实习岗位验证机制；
  • 二期：引入更多雇主，形成真正的双边网络。

原则 4：HR / 法务是共同 owner，不是配合方

公司挂名、招真人实习生，涉及候选人数据隐私、招聘公平性、岗位真实性。

必须把招聘团队 / HRG、法务在立项阶段就拉进来当合伙人——既是合规需要，也是给项目和个人上保险。不虚假承诺岗位、不滥用候选人数据。

五、玩法机制设计（让活动本身成为产品 demo）

5.1 报名即 demo、报名即筛选

入口设计为 「让你的龙虾来帮你报名 / 完成一道实战任务」：

  • 天然筛掉不 AI native 的人（门槛即过滤器）；
  • 顺手展示了产品能力；
  • 采集「谁真的会用 AI」的高质量信号，供企业侧精准锁定。

5.2 双 Agent 撮合可视化

把「招聘龙虾」与「求职龙虾」的破冰、问答、匹配过程做得可看、可分享：

  • 对学生是有趣的产品体验；
  • 是天然的传播素材（学生爱晒「我的龙虾帮我聊到了面试」）；
  • 对企业是「AI 初筛过程透明可信」的体验。

5.3 人在关键节点回到中心

Agent 负责前 90%（筛选 / 包装 / 破冰），人负责最后 10%（真人面试、决策、录用）。

这正是觅游相对纯 A2A 的差异化——把人拉回关键决策节点。

5.4 撮合链路（一句话流程）

学生声明诉求 → 一键生成「求职龙虾」 → 龙虾包装简历/完成任务
        ↓
招聘龙虾持续匹配 + Agent 间破冰问答
        ↓
匹配达标 → 推送给真人 HR → 真人面试 → Offer
        ↓
活动后：求职龙虾持续为该学生匹配新机会（留存）

六、北极星指标

指标类别具体指标说明
撮合效果撮合成功率推进到面试 / Offer 的比例（核心）
产品假设代理人托管率报名用户中真正生成并使用「求职龙虾」的比例
留存（关键）活动后复用率活动结束后「求职龙虾」继续被使用的比例
双边满意学生 / 企业满意度双边对撮合质量的评分
规模（参考）参与人数 / 岗位数仅作传播参考，不作为成败判据

从第一天起就把「会不会留」当成衡量项盯死，而非只看活动当天的峰值。

七、范围控制：暑期可落地的 MVP

暑假很近，范围要狠狠收窄，先验证假设、别陷进工程交付：

  • 撮合链路只跑通核心一环（Agent 初筛 + 破冰 + 推送真人）；
  • 其余环节用人工补位（concierge MVP，即「人肉跑通流程」也接受）；
  • 先做小批量（如限定 N 个岗位、定向高校 / 社群），跑通后再放量。

九、风险与应对

风险应对
办成一次性烟花、活动后即凉强制「活动后留存」设计（原则 2）；指标盯「复用率」
只是普通招聘会、与产品无关强制「Agent 撮合」机制（原则 1）；报名即 demo
供给侧单一、撑不起网络叙事明确分期；一期美团锚定、二期引雇主
合规 / 品牌风险HR + 法务共同 owner；不虚假承诺、不滥用数据
暑期时间紧、范围失控MVP 化 + concierge 补位 + 小批量先跑

十、下一步建议

  1. 立项对齐：与招聘团队、法务拉一次会，确认岗位资源与合规口径，确立共同 owner。
  2. 定义 MVP 链路与指标：把第五、六节细化为可执行的流程与可埋点的指标。
  3. 小批量试跑：定向某高校 / 社群，跑通核心撮合环节，验证「托管率 + 撮合成功率」。
  4. 再放量引爆：试跑数据达标后，配合增长 / 市场做大规模传播。
